# Разработка модуля семантического анализа текстов для ИИ-ассистента преподавателя

## Назначение проекта

Проект реализует модуль, который анализирует структуру и смысл учебных работ (курсовых) и выявляет потенциальные логические проблемы:

- слабая связность между соседними частями;
- несогласованность введения/заключения с основной частью;
- избыточное повторение смыслов.

Ключевая идея: эмбеддинг-модель используется как инструмент представления смысла, а интеллектуальное ядро проекта — это интерпретируемые метрики и правила анализа.

## Текущий статус по этапам

### Этап 0. Предобработка данных — выполнен

Реализовано:

- парсинг PDF и DOCX (`parser.py`);
- извлечение структуры документа (разделы/подразделы, номера, уровни);
- очистка текста (номера страниц, подписи/ссылки на рисунки, шум);
- исключение нерелевантных разделов (`Список литературы`, `Литература`, `Приложение`);
- разворачивание в плоские сегменты с ролями `intro/main/conclusion/other` (`preprocess_sections.py`);
- формирование итогового JSON с `segments` (`result.json`).

### Этап 1. Формирование эмбеддингов — выполнен

Реализовано:

- единый модуль расчёта эмбеддингов (`compute_embed.py`);
- поддержка двух режимов:
  - `SentenceTransformer`-модели;
  - обычные BERT-модели через mean-pooling;
- L2-нормализация векторов;
- сохранение:
  - `out/meta.json`;
  - `out/emb_<tag>.npy`.

Текущий набор моделей:

- `ai-forever/FRIDA`;
- `ai-forever/sbert_large_nlu_ru`;
- `ai-forever/ruBERT-base`;
- `cointegrated/rubert-tiny2`;
- `sentence-transformers/paraphrase-multilingual-mpnet-base-v2`;
- `sentence-transformers/all-mpnet-base-v2`.

### Этап 2. Сравнение моделей и выбор рабочей — выполнен

Реализовано в `versus_models.ipynb`:

- `sim_<tag>.npy` (полная cosine-матрица);
- `adj_<tag>.npy` (соседние пары);
- метрики `Cohen's d`, `KS`, `delta`, `intro/main`, `conclusion/main`, `redund_90`;
- multi-seed устойчивость (`mean/std`);
- сводный артефакт `out/metrics_summary.json`.

Итог этапа: по текущему корпусу и протоколу лучшей выбрана **FRIDA**.

Важно: это не «абсолютно лучшая модель вообще», а лучшая в рамках текущих данных и методики.

### Этап 3. Анализ документа и выявление логических проблем — в реализации

Этап 3 — основной смысл проекта: применение выбранной модели для диагностики конкретной работы и формирования интерпретируемого отчёта.

#### История итераций Этапа 3

- `compute_metrics.ipynb` — ранний базовый прототип (legacy).
- `compute_metrics_v1.ipynb` — зафиксированная версия baseline.
- `compute_metrics_v2.ipynb` — расширение набора метрик (drift/centrality/coverage/novelty).
- `compute_metrics_v2_1.ipynb` — текущая рабочая версия.

#### Что реализовано в `compute_metrics_v2_1.ipynb`

Слой 1. Диагностические метрики:

- локальная связность (`adj_mean`, слабые переходы);
- согласованность структуры (`intro/main`, `conclusion/main`, `intro_delta`, `conclusion_delta`);
- избыточность (`redund_90`);
- `semantic_drift_score`;
- `smoothness_score`;
- `segment_centrality` и `outlierness` (без self-similarity в диагонали);
- `coverage_score`;
- `novelty_redundancy_profile` (`redundant` / `jump` / `balanced`).

Слой 2. Интерпретационные правила:

- rule-based флаги проблем по порогам.

Слой 3. Финальный отчёт:

- структурированный JSON-отчёт с деталями метрик и флагов;
- агрегированный `risk_score_v2_1` как вспомогательная итоговая оценка.

Текущий выходной файл этапа:

- `out/document_metrics_report_v2_1.json`.

## Ограничения

1. Парсинг PDF зависит от качества страницы содержания.
2. Для DOCX ожидаются корректные стили `Heading 1/2` (`Заголовок 1/2`).
3. Этап 3 пока реализован в формате ноутбука (нет полноценного CLI-пайплайна).
4. Нет зафиксированного `requirements.txt`/`pyproject.toml` с версиями.
5. Выбор FRIDA и текущие пороги валидны для текущего корпуса; при расширении корпуса нужна повторная проверка.
