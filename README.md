# Разработка модуля семантического анализа текстов для ИИ-ассистента преподавателя

## Назначение проекта

Проект реализует модуль, который анализирует структуру и смысл учебных работ (курсовых) и выявляет потенциальные логические проблемы:

- слабая связность между соседними частями;
- несогласованность введения/заключения с основной частью;
- избыточное повторение смыслов.

Ключевая идея: модель эмбеддингов используется как инструмент представления смысла, а ядро диплома - это метод анализа (метрики + правила интерпретации).

## Текущий статус проекта

### Этап 0. Предобработка данных - выполнен

Реализовано:

- парсинг PDF и DOCX (`parser.py`);
- извлечение структуры документа (разделы/подразделы, номера, уровни);
- очистка текста (номера страниц, подписи/ссылки на рисунки, шум);
- исключение нерелевантных разделов (`Список литературы`, `Литература`, `Приложение`);
- разворачивание в плоские сегменты с ролями `intro/main/conclusion/other` (`preprocess_sections.py`);
- формирование итогового JSON с `segments` (`result.json`).

### Этап 1. Формирование эмбеддингов - выполнен

Реализовано:

- единый модуль расчёта эмбеддингов (`compute_embed.py`);
- поддержка двух режимов:
  - `SentenceTransformer`-модели;
  - обычные BERT-модели через mean-pooling;
- L2-нормализация векторов;
- сохранение:
  - `out/meta.json`;
  - `out/emb_<tag>.npy`.

Текущий набор моделей в коде:

- `ai-forever/FRIDA`;
- `ai-forever/sbert_large_nlu_ru`;
- `ai-forever/ruBERT-base`;
- `cointegrated/rubert-tiny2`;
- `sentence-transformers/paraphrase-multilingual-mpnet-base-v2`;
- `sentence-transformers/all-mpnet-base-v2`.

### Этап 2. Семантические метрики - частично выполнен (в ноутбуке)

В `versus_models.ipynb` уже есть рабочие расчёты:

- полная матрица косинусной близости `sim_<tag>.npy`;
- локальная связность соседних сегментов `adj_<tag>.npy`;
- оценки:
  - `intro -> main`,
  - `conclusion -> main`,
  - избыточность `main-main` (доля пар выше порога),
  - сравнение `adjacent` против случайных пар через `Cohen's d` и `KS`.

Вывод: позиция проекта уже не «между этапами 0 и 1», а «этап 1 завершён, этап 2 в активной реализации».

## Как делать лёгкое дообучение (если решим включать)

Принцип:

- разметка `0/1` используется не как классификация, а как контрастивный сигнал для геометрии эмбеддингов;
- итоговые значения связности (диапазон 0..1) возникают после обучения через cosine similarity.

Минимально реалистичный сценарий:

1. Собрать пары полуавтоматически:
- `positive`: соседние логически связанные сегменты одной работы;
- `negative`: несвязанные фрагменты (в т.ч. из разных работ).

2. Проверить пары вручную (не размечать всё с нуля).

3. Дообучить базовую SBERT-модель и повторить те же метрики.

4. В дипломе показать сравнение `до/после` по одному протоколу.

## Идеи, которые не первоочередные, но полезные

- Визуализация карты связности документа (heatmap матрицы сходства).
- Автоматическая генерация «проблемных переходов» (top-N минимальных `adjacent`).
- Агрегированный скор документа с расшифровкой вклада каждой метрики.
- Небольшой веб-интерфейс для преподавателя (загрузка работы -> отчёт).

## Структура репозитория

- `parser.py` - парсинг PDF/DOCX и постобработка структуры.
- `preprocess_sections.py` - очистка и подготовка `segments`.
- `compute_embed.py` - расчёт и сохранение эмбеддингов.
- `make_embeddings.ipynb` - базовый сценарий генерации `result.json` и `out/*`.
- `versus_models.ipynb` - сравнение моделей и расчёт метрик.
- `result.json` - текущий пример подготовленных сегментов.
- `out/` - метаданные, эмбеддинги и промежуточные метрики.

## Форматы данных (текущие)

`result.json`:

- верхний уровень: `title`, `author`, `type`, `segments`;
- сегмент: `id`, `parent`, `path`, `role`, `text`.

`out/meta.json`:

- для каждого сегмента: `id`, `role`, `path`, `parent`.

`out/emb_<tag>.npy`:

- матрица эмбеддингов формы `(N, D)`, где:
  - `N` - число сегментов;
  - `D` - размерность модели.

## Текущие ограничения

1. Парсинг PDF зависит от качества распознавания страницы содержания.
2. Для DOCX ожидаются корректные стили `Heading 1/2` (`Заголовок 1/2`).
3. Оценка пока запускается в ноутбуках; нет отдельного воспроизводимого CLI-пайплайна метрик.
4. Нет `requirements.txt`/`pyproject.toml` с зафиксированными версиями.
